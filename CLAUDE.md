# jobwatch — CLAUDE.md

Cloudflare Worker that polls company careers portals every 5 minutes and pushes a
Telegram alert when something the user can actually apply to opens. Two tracks:

- **new grad** — full-time entry-level SWE, class of May 2027
- **off-season internship** — Fall 2026, Winter, Spring 2027 (and co-ops)

**Summer internships are deliberately excluded.** The user graduates May 2027:
Summer 2026 is happening now, and Summer 2027 is after graduation. Neither is
something they can take. This is why `intern` in a title is neither a yes nor a
no on its own — it depends entirely on the term. See `src/classify.js`.

Runs on **Workers Paid ($5/mo)**, not the free tier. All-in cost is ~$5.40/mo;
`npm run budget` re-proves that against live boards whenever the list grows.

## Commands

```bash
npm run check                         # sync + probe + tests. Run before every deploy.
npm run sync                          # companies.txt -> src/registry.js + docs/UNRESOLVED.md
npm run probe                         # hit every configured adapter live
npm test                              # cron path: seeding, diffing, failure counter

npx wrangler dev --test-scheduled     # local; hit /__scheduled to fire the cron
npx wrangler deploy
npx wrangler tail                     # live logs
node scripts/discover.mjs stripe      # probe which ATS a company uses

curl "$WORKER/health?key=$ADMIN_KEY"        # adapter status
curl "$WORKER/run?shard=0&key=$ADMIN_KEY"   # force one shard
curl "$WORKER/test?key=$ADMIN_KEY"          # send a fake alert
```

Secrets live in `wrangler secret`, never in `wrangler.toml`:
`TG_TOKEN`, `TG_CHAT_ID`, `ANTHROPIC_KEY`, `ADMIN_KEY`.

## Layout

```
src/    engine.js      the FIELD-AGNOSTIC poll engine: createEngine({COMPANIES,
                         classify, track}) -> { checkCompany, runAll, digest,
                         handler }. Both Workers share this. Invariants live here.
        adapters.js    per-ATS fetchers -> {id,title,location,url}  (+ describe())
        dashboard.js   the application tracker: KV, JSON API, page, client, CSS
        index.js       SOFTWARE worker: engine bound to the software profile
        classify.js      software classifier: screen/classify/track + inUS + gate
        config.js        COMPANIES = software registry + aggregator
        registry.js      GENERATED by `sync`
        overrides.js     hand-maintained software overrides
        index.mech.js  MECHANICAL worker: engine bound to the mechanical profile
        classify.mech.js mechanical classifier (tight allowlist, new-grad only)
        config.mech.js   COMPANIES = mechanical registry, NO aggregator
        registry.mech.js GENERATED by `sync mech`
        overrides.mech.js hand-maintained mechanical overrides
companies.txt          software watchlist (SOURCE OF TRUTH)
companies-mech.txt     mechanical watchlist (SOURCE OF TRUTH)
wrangler.toml          software worker  (main = src/index.js)
wrangler.mech.toml     mechanical worker (main = src/index.mech.js)
scripts/               sync, probe, budget, discover, hunt (node, not the worker)
docs/                  capture.md, KICKOFF.md, UNRESOLVED{,.mech}.md (GENERATED)
test/                  node --test; no wrangler, no network
```

Everything under `src/` is worker code; `scripts/` are local dev tools that may
touch the filesystem — the worker never does.

## Two Workers, one engine

There are **two Workers on the same Cloudflare account**, one per person/field:
`jobwatch` (software, Ryan) and `jobwatch-mech` (mechanical, his friend). They
share every line of `engine.js`, `adapters.js` and `dashboard.js`, and differ only
in a **profile**: `{ COMPANIES, classify, track }` plus a watchlist. `index.js`
and `index.mech.js` are each ~5 lines that bind a profile to `createEngine`.

Isolation is total: **separate KV namespaces, separate Telegram chats, separate
`ADMIN_KEY`/dashboards, separate secrets.** One's broken adapter can't touch the
other. Cost stays ~$5/mo: Workers Paid is per-ACCOUNT, and KV/Anthropic usage sits
inside the same headroom (the mechanical classifier makes no LLM calls at all).

Anything field-specific belongs in the classifier, never in `engine.js`. The
scripts take a profile: `node scripts/sync.mjs mech`, `PROFILE=mech node
scripts/probe.mjs` (or `npm run sync:mech` / `probe:mech` / `check:mech`), which
read `companies-mech.txt` + `overrides.mech.js` and write `registry.mech.js` +
`UNRESOLVED.mech.md` — the software files are never touched.

The mechanical classifier is deliberately **pure regex** (tight allowlist: a mech
field signal AND an entry-level signal, software excluded, internships dropped).
No LLM, no description read — there's no SimplifyJobs-style aggregator for
mechanical roles to backstop a miss, so recall is the watchlist's job, and the
allowlist is widened by editing `MECH_FIELD` / `ENTRY_SIGNAL` in classify.mech.js.

## Architecture

```
companies.txt  → scripts/sync.mjs → src/registry.js → src/config.js   (build time)

cron (5 min) → for EVERY company, 6 at a time:
    adapters.js  → hit that company's portal, normalize to {id,title,location,url}
    KV diff      → drop IDs we've already seen
    inUS()       → drop confidently-foreign roles (free, before any LLM call)
    classify.js  → screen(title, {trusted}) → "high" | "maybe" | "no" | AMBIGUOUS
                     new grad : ALLOWLIST. entry-level signal in the title → high,
                                else "no". (aggregator = trusted → a bare title
                                is "maybe", not dropped.)
                     intern   : term decides. seasonless → AMBIGUOUS
                     └─ AMBIGUOUS only: describe() → read the posting
                        → tooMuchExperience()? → "no"
                        → else ask the LLM (title + requirements excerpt)
    index.js     → 🔴 loud push / 🟡 silent push / drop
```

**We are on Workers Paid ($5/mo), and that is load-bearing.** The free tier caps
CPU at 10ms per invocation, which is ~3 JSON boards — that is the *only* reason
sharding ever existed. Paid raises the cap to 30s (`[limits] cpu_ms` in
wrangler.toml), so we poll all 43 companies on every tick.

Do not reintroduce sharding. It would divide a fixed polling budget among more
companies and push per-company latency toward the ~30 min the aggregator already
gives you for free — which defeats the entire product. The bound on concurrency
(`POLL_CONCURRENCY = 6`) is politeness to the ATS endpoints, not CPU.

At 43 companies every 5 min we use roughly 1.5M KV reads and a few thousand KV
writes a month, against 10M and 1M included. Everything has ~5-10x headroom;
the $5 covers several hundred companies before anything binds.

## Invariants — do not break these

These are load-bearing. Each one was a bug or a deliberate call.

1. **KV writes and deletes are the scarce resource.** On Paid that's 1M/month
   each (it was 1,000/**day** on free). Reads are 10M/month and effectively free.
   NEVER write or delete unconditionally in the poll path — always read first,
   write only on real change. Two places this bites, both already handled and
   both covered by tests: a broken adapter must stop incrementing its counter
   once it has alerted (otherwise one dead adapter = 288 writes/day forever),
   and the digest is written **once per tick**, not once per company.

2. **`seen:{company}` is a cumulative union, never an overwrite.** Some adapters
   return a sliding window (Amazon uses `sort=recent&limit=100`). If we stored
   only the current page, an old job resurfacing would re-alert AND burn a write.
   Union with the previous set, cap at 3000.

3. **The classifier fails OPEN on ERRORS and on internships.** An Anthropic API
   error returns `"maybe"`, not `false`; a seasonless internship is escalated, not
   dropped. A false positive costs 2 seconds of scrolling; a false negative costs
   a job. **But the NEW-GRAD track is an allowlist, not fail-open** — see 3b.

3b. **New grad is an ALLOWLIST (title must name entry-level).** `NEWGRAD_SIGNAL`
   in classify.js: a full-time engineering title alerts only if it positively says
   entry-level (new grad, grad, junior, 2027, entry level, level 1, Engineer I/1,
   associate, apprentice…). A bare "Software Engineer" is **dropped**, not
   escalated. This was a deliberate reversal (the user was flooded with mid/senior
   roles): in practice a genuine new-grad role names itself in the title, so the
   miss rate is near zero. The safety net is the aggregator — `screen(t,
   {trusted:true})` keeps a bare title from the new-grad-CURATED SimplifyJobs feed
   (on the quiet channel), so a company that titles its new-grad role plainly is
   still caught there even though its own board drops it. Widen coverage by adding
   a phrase to `NEWGRAD_SIGNAL`, not by loosening this back to fail-open.

4. **Location IS a gate now — US-only.** (This reverses the original "location
   is a tag" rule, at the user's request.) `inUS()` in classify.js drops roles
   whose location is confidently non-US (Warsaw, Bengaluru, London, Manila...).
   It **fails open**: an empty location, a bare "Remote", or a US city with no
   state all pass — the gate only drops when it's *sure* it's foreign, so it
   never costs a real US job. Checked BEFORE classify(), so a dropped role
   doesn't even spend an LLM call. It's a blocklist of foreign places, not an
   allowlist of US ones, precisely so unknowns fail open.

4c. **The description is read ONLY for the ambiguous branch.** Since new grad is
   now a title allowlist (3b), the only titles that reach a description read are
   **seasonless internships** ("Software Engineer Intern", no term). There
   `classify()` calls `describe()` to catch an "intern" posting that actually
   wants years of experience. An explicit "New Grad"/"SWE I" is already `high`, a
   bare "Software Engineer" or "Senior Staff" is already `no`; none cost a request.
   Seeding never classifies at all. (Historically this gate also rescued bare
   full-time titles; the allowlist made that unnecessary — they're dropped now.)

   Lever and Ashby ship the description in their list payload (free). Greenhouse
   and Workday need a per-job detail fetch. Greenhouse's list *does* take
   `?content=true`, which would hand you every description in one call — it
   returns ~7MB for a big board, per poll. Don't. Everything else returns `""`,
   and `""` means the gate abstains: it **fails open**, like every other gate.

   `tooMuchExperience()` drops only on a confident signal: it takes the LOWEST
   year count stated anywhere ("1 required, 5 preferred" is a 1-year role), and a
   new-grad phrase anywhere in the description overrides it outright, because
   real new-grad reqs do list years under "preferred". Recall wins ties.

4b. **`intern` in a title is neither a yes nor a no.** Only the term decides.
   Fall 2026 / Winter / Spring 2027 → yes. Summer → no (see the two tracks
   above). No term named → ask the LLM, then fail open to the quiet channel.
   Check the engineering signal BEFORE the season, or "Marketing Intern, Fall
   2026" sails through on the season match alone. There is a test for that.

5. **A zero-jobs response counts as a FAILURE, not an empty result.** Silent
   adapter breakage is the worst outcome — the user assumes Amazon hasn't posted
   yet while the endpoint has been 404ing for three weeks. Track consecutive
   failures in KV, alert after 6.

6. **Nothing makes a sound. Everything still notifies.** `disable_notification:
   true` on EVERY message — 🔴, 🟡 and the ⚠️ breakage warning. The user asked
   for a notification without a sound, and Telegram has exactly two tiers (sound,
   or silent-with-notification). There is no third one.

   The cost: 🔴 and 🟡 are now identical in *delivery*. They differ only by emoji
   and the track tag in the message body. The original design leaned on sound to
   separate them — that lever is gone, deliberately. If high-confidence matches
   ever need to cut through again, flip `alertHigh` to `{ silent: false }`; there
   is nowhere else to go. A test asserts no path can reintroduce sound.

   What has NOT changed: nothing is ever silently dropped. Uncertain still gets
   delivered, and the daily digest still backstops a bad classifier verdict.

## Things NOT to build

- **Auto-apply / headless browser / Playwright form-filling.** Already evaluated
  and rejected. Every major ATS gates submission behind the *company's* private
  API key, so it means browser automation: CAPTCHAs, per-company Workday
  accounts, custom essay questions left blank, ToS violations, ban risk, and it
  isn't free. If asked, push back and point to the Simplify autofill extension
  plus a pre-staged answer bank instead.
- **Scraping Greenhouse's HTML.** `boards-api.greenhouse.io` is not an
  aggregator. Greenhouse is an ATS. Stripe's careers page renders from that exact
  endpoint. It IS the portal. Don't "upgrade" it to something slower.
- **A web dashboard for DISCOVERY.** The interface for *finding* jobs is
  Telegram. Don't build a web UI to browse open roles — that's what the alerts
  are. (This does NOT forbid the application tracker below, which is a different
  thing: a private record of jobs you've already applied to.)

## Application tracker (src/dashboard.js)

A private CRM for roles you've APPLIED to — deliberately separate from discovery
(above), which stays Telegram-only. Same Worker, same KV, same `ADMIN_KEY` gate.

Pipeline: `applied → oa → phone → onsite → offer / rejected / ghosted`. There is
**no "saved" stage** — a row exists because you applied to it. (One existed
briefly; it was removed. `normalize()` maps any leftover `saved` row, and any
other unknown status, to `applied` on read, so old data migrates itself without a
write.)

`STATUSES` is the single source of truth — each entry carries a `kind` (which
drives the header counts) and a `color` (which generates the CSS for the pill,
the card's left border, the kanban column and the menu swatch). Add a stage there
and the whole UI follows; there is no second list to update.

**Status is not the only axis.** `track` (new grad vs internship) is the other,
and it deliberately uses NO colour: the status already owns hue and the left
border. Internship cards are **dashed**, new-grad cards solid, and each carries a
text badge. Don't "improve" this by giving the track a colour — it collides with
the status palette, which is the mistake this avoids.

- `GET /dashboard?key=` — the page. List view (New Grad / Internships, split by
  `track()`) or a kanban board, drag to change status. Board cards are compact on
  purpose: company, track, role. The column already tells you the status, so a
  pill there would repeat the header you just read.
  Sort, filters, star to pin, per-card notes, inline edit (every field, incl.
  free-text comp), Export/Import JSON. Keys: `n` add, `/` search, `Esc` cancel.
- The page is served `cache-control: no-store`. It embeds your applications
  inline as JSON, so a cached copy is stale DATA, not just stale markup. Without
  it the browser and the CF edge will happily serve you yesterday's page after a
  deploy — which looks exactly like a broken deploy. It cost an hour once.
- `GET /applied?key=&j=` — one-tap logging. Every Telegram alert carries a
  "✅ Log as applied" link; tapping it files that exact role. The job is encoded
  inline in the link (no KV write at alert time). `id` is a hash of company+url,
  so re-tapping is a **no-op**: it will not reset a status you've advanced, and
  it does not touch `updatedAt` (that would reset the staleness clock on a role
  you haven't actually moved).
- `/api/app` — POST add (or restore a deleted row verbatim, which is what the
  undo toast sends) / PATCH status+notes+track+priority / DELETE.
  `/api/apps` GET lists. `/api/import` POST merges a backup, newest `updatedAt`
  wins so restoring an old export can't roll a live application back to Applied.
  `/api/parse?url=` GET guesses the company off a pasted posting URL.

Dates: `createdAt` is when the row appeared and `appliedAt` is when you applied
(they differ only for a hand-entered backdated row). `updatedAt` is the staleness
clock: an Applied/OA row untouched for `STALE_DAYS` (14) shows a "stale" flag. So
**nothing may touch `updatedAt` unless the row really changed** — `handleApi`
diffs before writing, which is also invariant 1 (the UI PATCHes on every status
pick, including picking the status a row is already in). The card labels the two
dates separately; they are different facts and merging them hides the follow-up.

Data lives in one KV key, `applications` (a JSON array). Low volume — a handful
of writes a day, nowhere near budget. Don't split it into per-app keys; the
whole-blob read is what makes the page one KV read. Rows written before a field
existed are backfilled by `normalize()` on **read** — never migrate with a write.

The browser half ships as a **string** (`CLIENT_JS`), not a function: bundling
would rewrite a real function's body and `.toString()` would serialize a
reference to a helper that doesn't exist in the page. So it must stay
backtick-free and `${}`-free — it lives inside a template literal. Nothing type-
checks it, so a test parses it with `new Function` to catch a syntax error before
it ships a dead page.

Note: the "Log as applied" link contains `ADMIN_KEY`. Fine in a private Telegram
chat; just don't forward an alert.

## Adapter tiers (adapters.js)

| Tier | How | Companies |
|---|---|---|
| 1 | Public ATS JSON — stable, set-and-forget | greenhouse, lever, ashby, smartrecruiters, workday |
| 2 | Undocumented endpoints the company's own SPA calls — **these drift** | amazon, apple, microsoft |
| 3 | `html` — regex job links out of raw HTML. Crude, always works | fallback |
| — | `aggregator` — SimplifyJobs firehose, catches companies not on the list | coverage net |

When a tier-2 adapter breaks: DevTools → Network → XHR → find the fat JSON
response → Copy as cURL → strip non-load-bearing headers → remap. See `docs/capture.md`.
Budget ~15 min every couple of months. This is expected maintenance, not a defect.

## Adding a company

**`companies.txt` is the source of truth.** Add a line, then:

```bash
npm run sync    # auto-detects the ATS, regenerates src/registry.js + docs/UNRESOLVED.md
npm run probe   # confirms it actually returns jobs
```

- `src/registry.js` is **generated** — never hand-edit it.
- `src/overrides.js` is **hand-maintained** and sync never overwrites it. Workday
  hosts, tier-2 portals, and known-dead diagnoses live there. An
  override always beats an auto-discovered entry.
- Anything sync can't resolve lands in `docs/UNRESOLVED.md` with the slugs it
  tried. Crack it via `docs/capture.md` and add an `OVERRIDES` entry.

Sync requires a board to return **≥5 jobs** before it believes it. A board
answering with 1-2 jobs is a name collision with someone else's tenant (Uber,
ByteDance and Visa all do this on SmartRecruiters) — wiring one up would track
the wrong company forever while looking perfectly healthy.

First poll for a new company seeds silently: it records what already exists,
without alerting and without calling the classifier. Alerts begin on poll two.

## Style

Plain JS, ES modules, no build step, no TypeScript, no framework. Keep it small
enough to read in one sitting. The user is a CS junior who knows this stack
(TS/React/Node/Postgres) — explain tradeoffs, don't over-explain syntax.
